# 设计操作系统 v0.1 基础约定

日期：2026-07-22  
状态：历史补档 / 待用户书面复核
规则来源：RULE-SRC-0001 brainstorming-spec-review-gate

## 1. 定位

v0.1 定义“设计操作系统”的初始边界。

它不是传统 GDD，不是灵感库，也不是任务看板。它是一套用于持续接收资料、分层推理、登记系统、管理状态、生成测试、阻止未定型设计污染实现层的工作系统。

目标：

```text
收纳已有资料和设计
区分定型、未定型、待发散内容
建立逻辑系统与纯数据系统的边界
为 UE 实现前的规则、数据、测试建立前置流程
```

## 2. 基本事实入口

所有资料必须先进入资料入口，不直接进入定型设计。

v0.1 的入口规则：

```text
原始资料保留来源
新资料标记未查阅
查阅后拆分事实、推导、冲突、问题、可沉淀内容
AI 推理不得伪装成用户原话
```

后续 v0.4.1 已将该入口细化为：

```text
00_raw_sources/
00_inbox/
```

## 3. 五个责任面

v0.1 初版使用五层模型；后续 v0.2 修正为“五个责任面”，避免误解为严格上下层。

初始定义：

```text
R0 体验锚点
R1 元系统 / 流程产生器
R2 关系模型 / 对象关系图
R3 概率数值
R4 实现适配 / UE
```

规则：

```text
R0 记录体验目标、禁区、高光和验收信号
R1 记录规则模具、流程产生器和约束外壳
R2 记录类型化关系图、全局可能性和局部可达性视图
R3 记录随机变量、分布、稳定性和仿真指标
R4 记录 UE、输入、表现、资产绑定和工程测试
```

## 4. 定型权

不同责任面的定型权不同：

```text
R0：用户确认 + 证据
R1：AI 推理 + 用户批准
R2：一致性审查 + 关键项用户批准
R3：仿真指标裁决
R4：工程测试裁决
```

设计上层需求不能由 AI 单独定型。底层数值、概率、测试、工程实现可以更多依赖推理和测试，但仍需回链到规范。

## 5. 资源定义

资源统一定义为：

```text
Resource Value = 对玩家未来行动集合的改变
```

该定义覆盖：

```text
物资
经济单位
道具
空间机会
时间窗口
信息
玩法
玩家技术
系统知识
```

每类资源必须保留独立失败模式，禁止资源同质化。

## 6. 随机是一等系统

随机不是简单爆率字段。

每个随机变量至少声明：

```text
抽样层级
分布类型
稳定性
条件修正
观测指标
```

允许描述：

```text
随机种子
纯随机
单局固定概率
概率的概率
离散条件增益
连续条件增益
稳定性与跨参数一致性
```

## 7. 图模型

对象、资源、经济单位、交互物和流程关系使用全局类型化关系图表达。

原则：

```text
全局关系图是静态可能性真相
局部可达性是派生视图
路径树只在仿真、规划、回放或调试时有限展开
```

后续可升级为类型化关系图或超图，以表达多输入、多输出、条件和概率转换。

## 8. 生成器契约

底层系统不是固定流程，而是流程产生器。

每个系统需要契约化：

```text
输入状态
输出变化
硬约束
软目标
随机源
测试指标
失败模式
反模式 / 禁止特征
```

## 9. 验证策略

默认验证策略：

```text
大样本仿真优先
指标审查
最小人工案例
不针对单个失败样本定向调参
先判断失败属于哪个责任面
```

复杂度门槛：

```text
可读：硬门槛
可利用：硬门槛
可惊喜：软目标
```

## 10. 当前限制

v0.1 只定义设计操作系统的基础约定，不定义：

```text
具体卡片模板
结构化规格
测试报告闭环
raw source custody
UE 实现结构
```

这些由 v0.2-v0.5 逐步补充。
