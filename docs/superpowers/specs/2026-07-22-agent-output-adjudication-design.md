# 大模型讨论输出裁决方案

日期：2026-07-22  
状态：已批准设计，待书面复核
规则来源：RULE-SRC-0001 brainstorming-spec-review-gate
裁决规则来源：RULE-SRC-0002 tempfolder-agent-output-adjudication
作用域：仅限 TempFolder/ 下七个模型目录的大模型讨论输出裁决；不适用于 game_design_archive 主工作流。

## 1. 目标

对 `TempFolder/` 中每个模型目录的分析讨论进行独立裁决，判断其分析思路和实质观点是否合理，并将可继承但边界不清的观点改写为更严谨的命题。

本阶段不把用户最初的 19 条讨论内容视为标准答案。裁判必须自行分析前提、推导、替代解释、反例、适用边界和系统后果，目标是为后续形成高质量方案提供可信的论证零件。

## 2. 本阶段范围

### 2.1 包含

- 按模型目录审查目录内全部 Markdown 文件。
- 区分分析正文、原始讨论归档、流程方案和重复副本。
- 对有实质影响的命题逐项裁决。
- 严格区分逻辑错误与必要遗漏。
- 提取可继承资产。
- 对有限成立或表达失真的观点给出裁判改写。
- 对尚不能下结论的观点给出验证条件。
- 在每个模型目录新增一份 `裁判意见.md`。

### 2.2 不包含

- 不做跨模型总结、总榜或排名。
- 不生成最终游戏设计方案。
- 不把重复观点的出现次数当作正确性证据。
- 不修改、移动、删除或改名任何原始文件。
- 不因观点符合当前项目偏好而直接判为成立。
- 不用单一总分压缩不同维度的质量差异。

## 3. 裁决单位与目标目录

最小裁决单位是模型目录，不是单个文件。目录内各文件共同构成该模型输出的证据集。

当前目标目录：

- `TempFolder/DouBaoPro/`
- `TempFolder/DoubaoTurbo/`
- `TempFolder/DS/`
- `TempFolder/DS1/`
- `TempFolder/Gemini/`
- `TempFolder/GPT5_5/`
- `TempFolder/OpenRouter/`

每个目录新增：

```text
裁判意见.md
```

## 4. 证据与重复内容规则

### 4.1 证据分型

每个源文件必须先归入以下一种类型：

- `分析正文`：模型对问题的实质分析、推导或方案。
- `原始讨论归档`：主要承担上下文保存，不将用户原话计为模型贡献。
- `流程方案`：收纳协议、状态标记或协作流程等元过程设计。
- `重复副本`：与其他文件字节级一致的内容。

一个文件可同时包含多个部分。裁判需按段落说明实际审查范围，不能因标题而一刀切。

### 4.2 重复识别

- 使用 SHA-256 识别字节级重复内容。
- 重复目录仍各自生成完整 `裁判意见.md`，并记录相同哈希及对应副本。
- 重复内容在后续综合阶段只能算一条独立证据。
- 多次出现不构成多模型共识，也不提高命题真值或置信度。
- 本阶段只标记精确重复；语义近似但文本不同的内容仍独立裁决。

### 4.3 已知精确重复关系

- `DS/00_架构讨论_20260722.md` 与 `DS1/00_架构讨论_20260722.md`
- `DS/01_初步讨论分析.md` 与 `OpenRouter/00_初步讨论分析.md`
- `DS/INBOX_收纳区.md` 与 `OpenRouter/01_收纳区_INBOX.md`

## 5. 裁决方法

采用“命题级法证裁决为主，高杠杆命题做对抗式复审”的方法。

### 5.1 命题拆分

将输出拆为以下五类原子命题：

1. `复述`：对用户输入或问题状态的重述。
2. `推导`：由前提导出中间判断或结论。
3. `主张`：关于系统本体、设计原则或因果关系的判断。
4. `方案`：具体结构、机制、流程或工程建议。
5. `纪律建议`：质量门、验证方法、知识管理或开发约束。

命题拆分遵守两个边界：

- 不能把多个依赖不同前提的结论塞进一个命题。
- 不能把句子拆到失去可独立判断的语义。

### 5.2 基础审查

每个有实质影响的命题至少检查：

- **忠实性**：是否把探索性想法偷换成确定结论，或把裁判自己的观点归给用户。
- **类型边界**：是否混淆静态数据、运行时状态、逻辑规则、玩家能力、体验目标、验证工具和表现层。
- **推理有效性**：结论是否由前提推出，是否存在类比跳跃、循环论证、术语包装或假二分。
- **事实前提**：经验判断、竞品判断或工程判断是否有足够依据。
- **替代解释**：是否存在解释力相近但假设更少、成本更低的模型。
- **可证伪性**：是否定义输入、输出、约束、失败条件或观测指标。
- **系统后果**：是否引入组合爆炸、黑箱、无脑最优解、错误依赖、通胀或过度冻结。
- **增量价值**：相对原始讨论是否真正减少不确定性或扩大后续可行动集合，而非只更换分层术语。
- **工程时机**：观点是否正确但被过早工程化，例如无证据地引入 Director AI、DSL 或强制 DAG。

### 5.3 高杠杆命题的对抗式复审

满足任一条件的命题属于高杠杆命题：

- 会改变项目公理、核心模型或系统边界。
- 会排除大量后续设计路径。
- 会引入高成本基础设施或不可逆依赖。
- 会被多个后续系统共同引用。
- 一旦错误会污染仿真、经济、随机或玩家技术模型。

复审顺序固定为：

1. **钢人化**：给出该命题最合理、最强的解释。
2. **最强反例**：构造能直接挑战其必要性、充分性或适用域的反例。
3. **边界判断**：确定反例击穿的是核心命题还是其过度表述。
4. **修正命题**：保留有效内核，缩小或重写适用域。

## 6. 裁决标签

每个实质命题只能使用以下一个主标签：

- `[成立]`：前提和推导成立，概念边界足够清楚。
- `[有限成立]`：核心有价值，但必须限定适用域、条件或强度。
- `[假说待验证]`：逻辑可以成立，但依赖尚未证实的经验事实或效果判断。
- `[不成立]`：推理链、概念边界、事实前提或内部一致性存在实质错误。
- `[不可裁决]`：定义、上下文或证据不足，无法诚实下结论。

每项裁决附置信度：

- `[高置信度]`：结论主要依赖定义、形式逻辑、明确文本或可直接复核的事实。
- `[中置信度]`：结论合理，但仍依赖部分设计语境或未实测前提。
- `[低置信度]`：存在多种竞争解释，只能提供暂定判断。

置信度描述的是裁判判断的稳定性，不描述原命题的重要性。

## 7. “错”与“缺”的边界

### 7.1 错误

只有满足以下至少一项才进入“错误清单”：

- 前提与结论之间不存在有效推导。
- 概念发生偷换、混淆或类别错误。
- 同一输出内部自相矛盾。
- 把可能性说成必要性，或把相关性说成因果性。
- 方案违反其自身声明的目标或约束。
- 依赖可直接证伪的事实前提。

### 7.2 遗漏

只有满足以下条件才进入“遗漏清单”：

- 原分析的目标若要成立或落地，该环节是必要条件。
- 原输出没有覆盖该环节。
- 缺失本身不证明已有命题为错。

不能因为裁判想到了更多内容，就把所有未提及内容都列为遗漏。遗漏必须说明“缺少它会导致什么具体断裂”。

## 8. 单份裁判文件结构

每份 `裁判意见.md` 使用以下固定结构：

```markdown
# <模型目录名> 裁判意见

规则来源：`RULE-SRC-0002 tempfolder-agent-output-adjudication`

作用域：仅限 2026-07-22 `TempFolder/` 七个模型目录的大模型讨论输出审查；不属于 `game_design_archive` 主工作流评审机制，也不定义系统卡定型、测试闭环或 UE 实现规则。

## 1. 裁判范围与证据
## 2. 总体诊断
## 3. 命题裁决
## 4. 主要错误
## 5. 关键遗漏
## 6. 可继承资产
## 7. 裁判修正版命题
## 8. 待验证项
## 9. 重复与归因说明
```

### 8.1 命题编号

- 每个目录独立使用 `C01`、`C02`、`C03` 递增编号。
- 编号对应命题，不对应段落。
- 同一命题在后文被引用时只使用原编号，不重复创建。

### 8.2 命题裁决字段

每项命题至少记录：

```text
编号：
类型：
原文位置：
原命题：
裁决：
置信度：
理由：
系统后果：
处理去向：
```

`处理去向` 只能为：

- `直接继承`
- `改写后继承`
- `进入验证池`
- `拒绝`
- `仅保留为讨论史`

### 8.3 裁判新增内容的归因

裁判提出的反例、补充前提、边界或新方案必须明确标注：

```text
裁判新增：
```

禁止将裁判改写后的高质量命题反向归功给原模型。

## 9. 输出质量门

单份裁判文件完成前必须满足：

1. 每项实质批评都有可定位的原文证据。
2. 每项 `[不成立]` 都指出具体断裂位置。
3. 每项 `[有限成立]` 都写明成立边界。
4. 每项 `[假说待验证]` 都写明需要什么证据。
5. 每项修正版命题都说明改变了哪个前提、定义或适用域。
6. “错误”和“遗漏”没有混写。
7. 用户原话、模型观点和裁判新增观点完成归因隔离。
8. 没有因为术语复杂、数学化或工程化而自动提高评价。
9. 没有因为观点符合项目偏好而跳过反例检查。
10. 没有将重复副本计为独立支持。
11. 原始文件保持零修改。
12. 七个目标目录均存在且仅新增一份 `裁判意见.md`。

## 10. 执行顺序

1. 建立全量文件清单与 SHA-256 清单。
2. 标记文件类型和精确重复关系。
3. 按目录逐份读取全部证据。
4. 拆分并编号有实质影响的命题。
5. 执行基础审查。
6. 对高杠杆命题执行对抗式复审。
7. 写入该目录的 `裁判意见.md`。
8. 对该裁判文件执行质量门检查。
9. 完成七个目录后，只做覆盖率、重复关系和原文件零修改校验。

第 9 步不输出跨模型总结，仅验证裁判工作是否完整。

## 11. 验收标准

本阶段完成的判据：

- 七个模型目录均有独立的 `裁判意见.md`。
- 每份裁判覆盖目录内全部文件，并标明实际证据范围。
- 精确重复关系被显式记录，且未被当作多模型共识。
- 所有高杠杆命题均有钢人化、最强反例和边界重写。
- “成立、有限成立、待验证、不成立、不可裁决”使用一致。
- “错”与“缺”严格分离。
- 有价值但边界不清的观点已产生可继承修正版。
- 不存在总分、排名、跨模型总结或最终方案。
- `TempFolder/` 中所有原始文件内容保持不变。

## 12. 已知限制

- 当前工作区不是 Git 仓库，无法提交设计文档或使用版本历史证明原文件未变；执行时应以执行前后的 SHA-256 对照替代。
- 当前输入主要是同一轮讨论的模型输出，不能仅凭文本判断实际游戏效果；涉及体验、平衡和长期可玩性的命题只能进入验证池。
- 字节级去重不能识别改写后的语义重复；语义重复关系留待后续综合阶段处理，不能在本阶段据此统计共识。
